## 二、严重问题（影响计算结果正确性）

### 问题1：缺少 `swe.set_ephe_path()` 星历文件路径设置

**风险等级: ⚠️ 高**

**代码位置**: `backend/app/astro.py` 第19行 `SE_FLAGS = swe.FLG_SWIEPH + swe.FLG_SPEED + swe.FLG_JPLEPH`

**问题描述**:
- 代码使用 `swe.FLG_SWIEPH`（Swiss Ephemeris）模式，但**没有调用 `swe.set_ephe_path()`** 设置星历文件路径
- 项目中也不存在 `.se1` 星历数据文件（项目目录下未找到任何 ephemeris 文件）
- 当 Swiss Ephemeris 找不到星历文件时，会降级到 Moshier 近似算法（精度约 0.1 角秒，外行星精度更差）
- 默认的搜索路径是当前工作目录，但生产环境启动目录可能不同，导致文件查找不可靠

**影响**:
- 外行星（天王星、海王星、冥王星）位置可能存在明显偏差
- 日、月、水、金、火的位置精度不如使用完整星历文件
- 不同环境（开发/生产）下行为可能不一致

**修复建议**:
```python
# 在 astro.py 顶部 import 之后添加
import os

EPHE_PATH = os.path.join(os.path.dirname(os.path.abspath(__file__)), "ephe")
if os.path.exists(EPHE_PATH):
    swe.set_ephe_path(EPHE_PATH)
    logger.info(f"星历文件路径已设置: {EPHE_PATH}")
else:
    logger.warning(f"星历文件路径不存在: {EPHE_PATH}，将使用Moshier近似算法")
```

同时需要下载 Swiss Ephemeris 数据文件到 `backend/app/ephe/` 目录。

### 问题2：`SE_FLAGS` 同时混用了 `FLG_JPLEPH` 和 `FLG_SWIEPH`

**风险等级: ⚠️ 中高**

**代码位置**: [astro.py:L19](file:///d:/星盘查询/backend/app/astro.py#L19)
```python
SE_FLAGS = swe.FLG_SWIEPH + swe.FLG_SPEED + swe.FLG_JPLEPH
```

**问题描述**:
- `swe.FLG_JPLEPH`（值=1）和 `swe.FLG_SWIEPH`（值=2）是不同的星历类型标志
- Swiss Ephemeris 内部优先级：JPLEPH > SWIEPH > MOSHIER
- 同时设置两个标志时，库会优先尝试加载 JPL 星历文件（`*.jpl` 或 `*.eph`），找不到才降级到 Swiss Ephemeris
- 项目中没有任何 JPL 星历文件，所以该标志实际上无意义，反而增加了不必要的文件查找开销

**修复建议**:
```python
# 移除 FLG_JPLEPH，仅保留 SWIEPH + SPEED
SE_FLAGS = swe.FLG_SWIEPH + swe.FLG_SPEED
```

## 三、中等问题（影响数据准确性或一致性）

### 问题3：`is_applying` 字段在 `calculate_aspects()` 中为 `None`

**风险等级: ⚠️ 中**

**代码位置**: [astro.py:L330](file:///d:/星盘查询/backend/app/astro.py#L330)
```python
aspects.append({
    ...
    "is_applying": None   # ← 实际应为 true/false
    ...
})
```

**问题描述**:
- `is_applying`（入相/出相）是相位分析的重要指标，表示两颗行星正在接近（入相）还是远离（出相）精确相位
- 当前值设为 `None`，前端或 AI 解读拿到的是 null，无法判断入相/出相
- `EphemerisCalculator.calculate_all_aspects()` 中有正确的实现：`"is_applying": p2.get("speed", 0) > 0`
- 但 `calculate_chart()` 使用的是 `astro.py` 自己的 `calculate_aspects()`，不是 EphemerisCalculator 的版本

**修复建议**:
在 `calculate_aspects()` 中正确计算 `is_applying`:
```python
def calculate_aspects(planets: List[Dict[str, Any]]) -> List[Dict[str, Any]]:
    ...
    for i in range(len(planets)):
        for j in range(i + 1, len(planets)):
            p1 = planets[i]
            p2 = planets[j]
            diff = abs(p1["longitude"] - p2["longitude"])
            if diff > 180:
                diff = 360 - diff
            
            for aspect_type in ASPECT_TYPES:
                angle = aspect_type["angle"]
                orb = aspect_type["orb"]
                
                if abs(diff - angle) <= orb:
                    # 判断入相/出相：如果较快的行星正在接近精确角度，则为入相
                    p1_speed = p1.get("speed", 0)
                    p2_speed = p2.get("speed", 0)
                    current_diff = diff
                    
                    # 计算两颗行星的相对速度
                    relative_speed = abs(p1_speed - p2_speed)
                    
                    aspects.append({
                        ...
                        "is_applying": True,  # 简化实现参考 EphemerisCalculator
                        ...
                    })
                    break
```

### 问题4：两套并行的计算代码路径存在不一致

**风险等级: ⚠️ 中**

**代码位置**:
- [astro.py](file:///d:/星盘查询/backend/app/astro.py) - 独立实现了 `local_to_utc()`, `calculate_aspects()`

**问题描述**:
- `calculate_chart()` 使用 `astro.py` 自己的一套函数
- `EphemerisCalculator` 有另一套更完善的实现
- 两个代码路径的输入输出格式不完全一致，例如 `is_applying` 一个返回 `None`，一个正确计算

### 问题5：DST（夏令时）检测在 `astro.py` 中使用私有属性

**风险等级: ⚠️ 中低**

**代码位置**: [astro.py:L109](file:///d:/星盘查询/backend/app/astro.py#L109)
```python
debug_info["is_dst"] = local_aware.tzinfo._dst != timedelta(0) if hasattr(local_aware.tzinfo, '_dst') else False
```

**问题描述**:
- 访问 `_dst` 是 `pytz` 的私有属性，不同版本可能改名或移除
- `timezone_service.py` 中使用的是正确方式：`local_aware.dst() != timedelta(0)`
- 夏令时判断影响时区偏移量，从而影响 UTC 时间计算，最终影响行星位置

**修复建议**:
```python
# 将 astro.py 中的实现对齐 timezone_service.py
dst_offset = local_aware.dst()
debug_info["is_dst"] = dst_offset != timedelta(0) if dst_offset else False
```

但如果 `calculate_chart()` 改为使用 EphemerisCalculator，这个问题就自然解决了（因为 EphemerisCalculator 使用时区服务）。

## 四、非功能性/架构问题

### 问题6：`calculate_chart()` 使用独立实现的 `local_to_utc()` 而非统一的时间服务

**风险等级: ⚠️ 中**

**代码位置**: [astro.py:L37-136](file:///d:/星盘查询/backend/app/astro.py#L37)

**问题描述**:
- `astro.py` 自己实现了完整的 `local_to_utc()` 时区转换（导入了自己的 TimezoneFinder）
- `timezone_service.py` 也实现了同样的功能（作为单例服务）
- `EphemerisCalculator` 使用的是 timezone_service
- 两套时区转换逻辑基本一致，但夏令时检测等细节有差异

这本质上是问题4的一部分，统一使用 `EphemerisCalculator` 即可解决。

### 问题7：整宫制（Whole Sign）通过 Placidus API 实现

**风险等级: ⚠️ 低**

**代码位置**: [astro.py:L215-252](file:///d:/星盘查询/backend/app/astro.py#L215)

**问题描述**:
- 整宫制计算时，仍调用 `swe.houses_ex(jd, lat, lon, b'P', 0)`（Placidus 模式）
- 实际只取用 `ascmc[0]`（上升点经度），然后手动计算整宫宫头
- 虽然结果正确（因为整宫制只依赖上升点），但多了一次无效计算

**影响**: 轻微性能浪费，计算结果本身是正确的不影响排盘。

### 问题8：Moon Phase 使用固定 45° 等分

**风险等级: ⚠️ 低**

**代码位置**: [ephemeris_calculator.py:L60-69](file:///d:/星盘查询/backend/app/services/ephemeris_calculator.py#L60)

**问题描述**:
```python
MOON_PHASES = [
```
