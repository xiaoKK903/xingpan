## Fix 1: `.env.example` API Key 泄露

### 原因

`backend/.env.example` 文件中包含了真实的 `DASHSCOPE_API_KEY` 值：

```
DASHSCOPE_API_KEY=sk-556b657fe9d745539045879218f51920
```

`.env.example` 文件一般作为模板提交到版本控制（.gitignore 中不会忽略 `.env.example`），而真实的 API Key 应该只存放在被 `.gitignore` 忽略的 `.env` 文件中。这个失误导致 API Key 对任何有仓库访问权限的人可见。

### 原理

- `.env` 文件被 `.gitignore` 忽略，不会提交到 Git 仓库
- `.env.example` 没有被 `.gitignore` 忽略，会提交到 Git 仓库
- 任何克隆该仓库的人都能看到这个真实的 API Key
- 他人可以使用该 Key 调用通义千问 API，产生费用

### 修复方案

1. 将 `.env.example` 中的真实 Key 替换为占位符 `your-dashscope-api-key-here`
2. 添加中文警告注释，提醒开发者不要将真实 Key 提交到版本控制
3. 真实的 `.env` 文件保留原 Key，不受影响（已在 `.gitignore` 中）

## Fix 2: API 拦截器返回逻辑错误

### 原因

文件：`frontend/src/api/index.js`

原代码：
```javascript
return data.data || data
```

使用 `||` 运算符处理返回值。当 `data.data` 为以下**假值（falsy）**时，会错误地返回整个 `data` 对象（完整的响应包络 `{code, message, data}`）：
- `null`（如删除接口返回 `data: null`）
- `0`（如数字类型返回值）
- `false`（如布尔类型返回值）
- `""`（如空字符串返回值）

这导致调用方收到非预期的数据结构，可能引发 `undefined` 访问错误或逻辑异常。

### 原理

JavaScript 中 `||` 运算符的短路规则：
- 左侧为 **truthy** 时，返回左侧值
- 左侧为 **falsy** 时，返回右侧值

`null`、`0`、`false`、`""` 都是 falsy 值，但它们是合法的 API 响应数据。

正确的做法是使用 `'data' in data` 运算符检查属性**是否存在**，而非判断值的真假。

### 修复方案

```javascript
return 'data' in data ? data.data : data
```

- `'data' in data`：检查 `data` 对象是否包含 `data` 属性（无论值是什么）
- 如果包含，安全地返回 `data.data`（即使是 `null`、`0`、`false`）
- 如果不包含（极少见的边缘情况），返回整个响应对象作为 fallback

## Fix 3: CSS 过度使用 !important

### 原因

文件：`frontend/src/styles/global.scss`

原文件 715 行中，几乎每一条 CSS 规则都使用了 `!important`。这是由于开发者直接覆盖 Element Plus 组件样式时，为了快速生效而采取的方式。

### 原理

`!important` 的问题：
1. **特异性战争**：一旦使用 `!important`，后续任何样式（包括组件内部的 scoped 样式）都必须用 `!important` 才能覆盖，形成恶性循环
2. **不可维护性**：当 Element Plus 升级时，`!important` 规则可能导致新版本样式无法正常显示
3. **组件隔离失效**：Vue 的 scoped 样式机制被 `!important` 绕过，组件的局部样式无法覆盖全局样式
4. **调试困难**：浏览器 DevTools 中 `!important` 规则难以追踪和调试

Element Plus 2.x 支持 CSS 变量（Custom Properties）主题化，这是官方推荐的定制方式：

```css
:root {
  --el-color-primary: #8b5cf6;
  --el-bg-color: #0a0a1a;
  --el-text-color-primary: rgba(255, 255, 255, 0.9);
}
```

### 修复方案

1. **保留 CSS 变量定义** — 它们是 Element Plus 官方推荐的主题化方式
2. **去除所有不必要的 `!important`** — Element Plus 的 CSS 变量机制已经处理了大部分样式覆盖
3. **使用 `--el-*` 组件级 CSS 变量** — 例如 `--el-input-bg-color` 替代硬编码的 `background: xxx !important`
4. **添加响应式设计** — 增加 `@media (max-width: 768px)` 平板端和 `@media (max-width: 480px)` 手机端的断点
5. **仅在少数必要处保留 `!important`** — 例如 Element Plus 内联样式无法被 CSS 变量覆盖的 `.el-dialog` 宽度

文件从 715 行减少到 566 行，`!important` 使用次数从约 200 次减少到 4 次。

## Fix 4: Element Plus 图标全局注册导致包体积过大

### 原因

文件：`frontend/src/main.js`

```javascript
import * as ElementPlusIconsVue from '@element-plus/icons-vue'
for (const [key, component] of Object.entries(ElementPlusIconsVue)) {
  app.component(key, component)
}
```

这段代码将 `@element-plus/icons-vue` 包中约 **200 多个**图标全部注册为全局 Vue 组件。

### 原理

1. **包体积膨胀**：Webpack/Vite 的 tree-shaking 无法移除通过 `import *` 和 `Object.entries()` 动态引用的模块，所有 200+ 图标都会被打包
2. **冗余注册**：项目中 34 个组件文件已经在各自文件中通过 `import { IconName } from '@element-plus/icons-vue'` 按需导入了具体使用的图标
3. **全局命名污染**：所有图标名都注册为全局组件，可能与其他组件重名

### 修复方案

**移除全局图标注册** — 删除 `main.js` 中的 `import *` 和注册循环。

34 个组件已经通过以下方式按需导入各自所需的图标：
```javascript
import { User, Search, Loading } from '@element-plus/icons-vue'
```

这些导入在 `<script setup>` 中声明后自动在模板中可用，无需全局注册。

## Fix 5: Redis 缓存重复实现

### 原因

项目中存在两套独立的 Redis 缓存实现：

| 文件 | 类型 | 依赖 | 使用者 |
|------|------|------|--------|
| `services/cache_service.py` | 同步 (sync) | `redis` 库 | `life_script_service.py`, `routers/life_script.py`, `routers/transit.py` |
| `redis_client.py` | 异步 (async) | `redis.asyncio` 库 | `main.py`（生命周期管理）, `online_user_cache_service.py` |

两套系统各自创建独立的 Redis 连接池，互不共享数据，造成资源浪费和维护负担。

### 原理

1. **连接池重复**：当 Redis 可用时，两个模块各自建立一个连接池，浪费系统资源
2. **数据不互通**：`cache_service.py` 缓存的数据，`redis_client.py` 无法访问，反之亦然
3. **同步阻塞**：FastAPI 是异步框架，`cache_service.py` 中的同步 Redis 调用会阻塞事件循环
4. **维护成本翻倍**：任何 Redis 相关的功能变更需要同时修改两个文件

### 修复方案

1. **移除 `RedisCache` 类**：删除 `cache_service.py` 中的 `RedisCache` 类（约 130 行），该类与 `redis_client.py` 功能完全重复
2. **`CacheService` 精简为纯内存缓存**：去除 Redis 连接初始化逻辑，只保留 `MemoryCache` 功能
3. **更新 `transit.py` 的缓存状态检查**：将 `cache_service.is_using_redis` 引用改为直接从 `redis_manager` 获取状态
4. **添加注释指引**：在 `CacheService` 文档字符串中提示生产环境应使用 `redis_client.py`

这样 `cache_service.py` 仅作为本地开发的内存缓存使用，生产环境的 Redis 缓存统一通过 `redis_client.py` 和 `online_user_cache_service.py` 实现。

## Fix 6: 时区处理不一致

### 原因

项目中有两种不同的 UTC 时间获取方式：

**方式 A（带时区信息）** — 8 个文件中分别定义：
```python
def _utc_now():
    return datetime.now(timezone.utc)
# 返回: datetime(2026, 5, 5, 12, 0, 0, tzinfo=timezone.utc)
```

**方式 B（无时区信息）** — 100+ 处使用：
```python
datetime.utcnow()
# 返回: datetime(2026, 5, 5, 12, 0, 0)  — 时间是 UTC 但无 tzinfo
```

### 原理

Python 中，datetime 对象分为两类：
1. **Naive（朴素）** — 无时区信息，由 `datetime.utcnow()` 生成
2. **Aware（感知）** — 有时区信息，由 `datetime.now(timezone.utc)` 生成

两者的比较和计算会抛出 `TypeError`：
```
TypeError: can't compare offset-naive and offset-aware datetimes
```

当使用方式 A 生成的 datetime 与数据库中存储的方式 B datetime 进行比较时，就会出现此错误。

此外，8 个文件各自定义相同的 `_utc_now()` 函数，违反 DRY（Don't Repeat Yourself）原则。

### 修复方案

1. **创建共享工具函数**：在 `app/utils/datetime_utils.py` 中定义唯一的 `utc_now()` 函数
2. **替换所有 `_utc_now()` 定义**：更新 8 个文件中的函数定义为导入共享版本
   - 已使用 `_utc_now` 的文件：`match.py`, `pk_system.py`, `common.py`, `time_capsule.py`, `models.py`, `activity_service.py` — 替换为 `from app.utils.datetime_utils import utc_now as _utc_now`
   - 定义了但未使用的文件：`user.py`, `topic_challenge.py`, `social_plaza.py` — 直接删除函数定义
3. **清理未使用的导入**：移除不再需要的 `timezone` 导入
